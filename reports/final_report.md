## 1. Постановка задачи

Разработка модели **детекции конца реплики** для голосовых AI-агентов.
Бизнес-проблема: голосовые боты должны отвечать без задержки и не перебивать
пользователя. Это требует понимания «закончил ли пользователь свою мысль».

## 2. Данные

- **MultiWOZ 2.2** (HuggingFace `pfb30/multi_woz_v22`) — 8 437 диалогов
  про бронирование. Использовано 2 000 диалогов → ~100 000 префиксов.
- **DailyDialog** (HuggingFace `roskoN/dailydialog`) — 1 000 бытовых
  диалогов для drift-анализа (out-of-domain).

Препроцессинг: каждая USER-реплика разрезается на префиксы методом
prefix sampling. Y=1 только для последнего префикса.

## 3. Признаки

| Признак | Описание |
|---|---|
| `length_words`, `length_chars` | Длина префикса |
| `ends_with_punct` | Заканчивается ли на `.?!` |
| `ends_with_prep` | Заканчивается ли предлогом |
| `has_question_word` | Содержит ли вопросительное слово |
| `last_pos` | POS-тег последнего слова (one-hot) |

## 4. EDA — ключевые находки

- **Дисбаланс классов:** 87% / 13% → главная метрика F1
- **Длина префикса** статистически значимо различается (t-test p<1e-100)
- **POS-теги** сильно разделяют классы: ADP, DET → не конец;
  PUNCT, INTJ → конец (Cramér's V = 0.56)
- **`ends_with_punct`** — самая сильная корреляция с целью (+0.42)
- Мультиколлинеарность: `length_words` ≈ `length_chars` (0.95)

## 5. Моделирование

Train/test split по `dialogue_id` через `GroupShuffleSplit` (без утечки).
Подбор порога классификации **на train** для каждой модели — без утечки.

| Модель | F1 | Precision | Recall | ROC-AUC | Threshold |
|---|---|---|---|---|---|
| Baseline | 0.77 | 0.66 | 0.91 | — | — |
| LogReg | 0.79 | 0.72 | 0.88 | 0.97 | 0.83 |
| LightGBM (default) | 0.82 | 0.77 | 0.88 | 0.98 | 0.88 |
| **LightGBM (Optuna)** | **0.83** | 0.78 | 0.88 | 0.98 | 0.74 |

Optuna: 30 trials, GroupKFold(n=3), maximize F1.
Главные гиперпараметры по importance: `learning_rate`, `num_leaves`, `min_child_samples`.

Feature importance LightGBM: `ends_with_punct` доминирует (1.7M vs 175k у следующего).

## 6. Drift анализ (DailyDialog)

| Метрика | MultiWOZ (in-domain) | DailyDialog (out-of-domain) | Δ |
|---|---|---|---|
| F1 | 0.83 | 0.77 | −0.06 |
| Precision | 0.78 | 0.69 | −0.08 |
| Recall | 0.88 | 0.88 | 0 |

PSI по признакам:
- **`last_pos`: 1.06** (сильный drift — другое распределение POS из-за тематики)
- `length_words`: 0.085 (умеренный)
- Остальные < 0.1 (стабильные)

**Главный инсайт:** PSI огромный, но F1 упал слабо — модель опирается на
устойчивый признак `ends_with_punct`. Если бы зависела от `last_pos` —
развалилась бы.

## 7. Deployment

- **FastAPI** сервис с `/predict` и `/health` endpoints
- **Pydantic** валидация + автогенерация Swagger UI
- **Input normalization** — добавление терминальной пунктуации по эвристике

- **Docker** для изолированного запуска (~1 ГБ образ)

```bash
docker run --rm -p 8000:8000 turn-detector:latest
```

## 8. Выводы

1. **Сильный baseline (F1=0.77)** на правиле по пунктуации — главная находка.
   Любая ML-модель должна с трудом его обходить.
2. **LightGBM tuned достиг F1=0.83** — на 6 п.п. лучше baseline.
3. **Drift на out-of-domain составил 6% F1** — приемлемо, но требует
   периодического переобучения и мониторинга PSI в продакшене.
4. **Над-зависимость от `ends_with_punct`** — известное ограничение,
   решается на уровне API через input normalization.
